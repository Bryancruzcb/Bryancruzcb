# Animated Profile README — Design

**Date:** 2026-07-19
**Repo:** `Bryancruzcb/Bryancruzcb` (GitHub profile repo)
**Status:** Approved, pending implementation plan

---

## 1. Goal

Add a terminal-styled animated header to the GitHub profile README, built from
self-contained animated SVGs generated by Python and refreshed daily by GitHub
Actions.

Adapted from [avivashishta.com — How I Built an Animated GitHub Profile
README](https://www.avivashishta.com/blog/build-animated-github-profile-readme.html),
with deliberate deviations documented in §6.

### Non-goals

- Rewriting or condensing the existing README prose. It stays byte-for-byte intact.
- Third-party stats services (readme-stats, shields-based counters). All art is
  generated in-repo and committed.
- JavaScript of any kind. GitHub strips `<script>` from READMEs.

---

## 2. Constraint that shapes everything

GitHub strips `<script>` and sanitizes nearly all inline CSS in READMEs, but it
**does** render SVGs embedded via `<img>` and plays their SMIL / CSS-keyframe
animations. Therefore all motion lives inside self-contained SVG files, and the
README only places them.

Corollaries, verified:

- Inline `style="..."` is stripped. The only vertical spacing GitHub honors is `<br>`.
- `<h1>` / `<h2>` draw a full-width underline rule. Use `<h3>` for prompt labels.
- A `<table>` is the only reliable way to put two images side by side.

## 4. Decisions and their rationale

### 4.1 Additive scope

The art is a header. Existing content is preserved verbatim beneath it. The
existing project write-ups are the profile's strongest asset for a job search;
ASCII art carries aesthetic signal, not substance, and must not displace them.

### 4.2 Heatmap retained despite sparse data — accepted tradeoff

Measured 2026-07-19 from the public endpoint:

| Metric | Value |
|---|---|
| Day cells in window | 365 (2025-07-20 → 2026-07-18) |
| Days with ≥1 contribution | **13** |
| Total contributions | 168 |
| Of those, private/restricted | 12 |
| Empty months | 11 consecutive (2025-07 → 2026-05) |

The source article's design assumes a dense graph (its author: 9,376
contributions). At 13/365 the heatmap renders as a near-empty year in the most
prominent position on a job-hunting profile.

**This was raised explicitly and the decision was to keep it**, on the reasoning
that the daily cron keeps it live, so sustained activity during the job hunt
fills it in over the coming months.

Mitigation built into the design: `WEEKS` is a module-level constant in
`render_heatmap_svg.py`. Narrowing to a denser rolling window (e.g. 12 weeks) is
a one-line change, not a rewrite. The stats footer (§5.5) is also framed on
streak and recent activity rather than the annual total.

### 4.3 Source photo

Real photo supplied: a front-lit classroom selfie, dark curly hair, bright
whiteboard background, black t-shirt.

Empirically validated 2026-07-19 with a Pillow-only spike:

- At default tone settings the face collapses into an unreadable dark mass —
  hair and skin both floor out on `%`/`@`.
- At `gamma≈0.55, contrast≈1.6` facial structure resolves: eyes, nose bridge,
  and mouth are legible.

Conclusion: the pipeline works on this photo, but **tuning is load-bearing and
photo-specific**. See §6.6.

### 5.1 `prep_photo.py` — local, once per photo

**In:** source photo. **Out:** `source-prepped.png` (grayscale).

1. **Normalize orientation first.** `ImageOps.exif_transpose()`, then strip EXIF.
   The supplied photo has **EXIF orientation 5**, which includes a *mirror* as
   well as a rotation. PIL and OpenCV disagree about applying EXIF automatically;
   normalizing once up front and discarding the tag prevents a
   double-applied-or-mirrored face downstream.
2. **Remove background** with `rembg`. Required, not optional — the whiteboard's
   marker writing and window blinds otherwise speckle the negative space.
3. **Boost local contrast** with OpenCV CLAHE. Global autocontrast is
   insufficient (validated by spike); local adaptation is what separates facial
   features from hair.
4. **Composite onto pure white** so background maps to the blank end of the ramp.

**Parameters** (all CLI flags with defaults): `--gamma` (default `0.55`),
`--contrast` (default `1.6`), `--crop L,T,R,B` as fractions (default
`0.10,0.02,0.92,0.80`, trimming the black t-shirt which otherwise merges with the
hair), `--clahe-clip`, `--preview`.

`--preview` prints the resulting ASCII directly to the terminal without writing
any SVG, so tuning is a seconds-long loop.

### 5.2 `make_ascii_svg.py` — local

**In:** `source-prepped.png`. **Out:** `bryan-ascii.svg`.

- Downsample to a character grid, `COLS = 100`, rows derived from the cropped
  aspect ratio times a `CHAR_ASPECT = 0.5` factor (glyph cells are ~2× taller
  than wide).
- Map brightness to the density ramp below — bright→sparse, dark→dense. The
  leading space clears background to nothing.

  ```python
  RAMP = " .`:-=+*cs#%@"
  ```

- **Monochrome**, single fill `#c9d1d9`. Per-character coloring is what makes
  most ASCII portraits look like static.
- Animation: each row wrapped in a horizontal clip that wipes left→right with a
  small block cursor riding the wipe edge, staggered top→bottom. Prints once and
  **freezes** — no loop.

### 5.3 `make_info_card.py` — local

**Out:** `info-card.svg`. Hand-authored neofetch-style panel, 490px wide.

Content (approved draft):

```
Now       CS @ San Jose State University
Seeking   New-grad / junior SWE — Java backend, full-stack, tooling
Stack     Java 21 · Spring Boot · JavaFX · React 19 · TypeScript · Python
Ships     CreatorFlow · second-brain-tools · signal-path
Habits    tests · CI on every push · honest error handling
Where     Bay Area, CA — onsite or remote
```

Keys `#7ee787`, values `#c9d1d9`, title bar `#58a6ff`. Each row fades and slides
in on a short stagger. `STATIC=1` env var emits a frozen frame for local preview.

Deliberately excludes GitHub stats — the heatmap covers those. This card carries
what numbers cannot.

### 5.4 `fetch_contributions.py` — **runs daily in CI**

**Out:** `data/contributions.json`.

Scrapes `https://github.com/users/Bryancruzcb/contributions` — public HTML, no
token, no GraphQL. Verified returning HTTP 200 with 365 day cells.

**Verified markup contract:**

```html
<td class="ContributionCalendar-day"
    data-date="2026-07-12"
    data-level="1"
    data-ix="51"
    id="contribution-day-component-0-51"></td>
<tool-tip for="contribution-day-component-0-51">1 contribution on July 12th.</tool-tip>
```

- `data-level` is `0`–`4` (five levels — see §6.4).
- **Exact counts are not on the `<td>`.** They live in the sibling `<tool-tip>`
  text, joined to the cell via `tool-tip[for] == td[id]`. Parse
  `"N contribution(s) on <date>."`, treating `"No contributions"` as 0.

Emits raw days plus derived stats: current streak, longest streak, best day,
active days in last 30, monthly totals.

**Hard validation gate — must fail loudly:** if fewer than 300 day cells parse,
or zero cells have a non-zero level, `sys.exit(1)`. Rationale in §6.3.

### 5.5 `render_heatmap_svg.py` — **runs daily in CI**

**In:** `data/contributions.json`. **Out:** `contrib-heatmap.svg`, 860px wide.

- `WEEKS = 53` module-level constant (see §4.2).
- 53×7 grid of rounded boxes.
- `PALETTE = ["#161b22", "#0e4429", "#006d32", "#26a641", "#39d353"]` — exactly
  five entries, indexed by `data-level` 0–4.
- Reveal: diagonal line-after-line slide-down via CSS keyframes, plays on load
  then **freezes**. No looping glow.
- `Less ░▒▓█ More` legend.
- Stats footer framed on streak / recent activity, not the annual total.

### 5.6 `.github/workflows/update-profile-art.yml`

```yaml
name: Update profile art
"on":
  schedule:
    - cron: "17 6 * * *"     # ~06:17 UTC daily
  workflow_dispatch: {}
permissions:
  contents: write
jobs:
  heatmap:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -r scripts/requirements.txt
      - run: python scripts/fetch_contributions.py
      - run: python scripts/render_heatmap_svg.py
      - uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: "chore: refresh contribution graph [skip ci]"
          file_pattern: "data/contributions.json contrib-heatmap.svg"
```

Only the heatmap is regenerated. Portrait and info card are static — rebuilt
locally only when the photo or details change.

### 5.7 `.gitignore`

Excludes `source-photo.*` and `source-prepped.png`. The ASCII output is an
abstraction of the face; the raw selfie is not, and this repo is public. The
committed SVGs are the deliverable — a fresh clone does not need to reproduce
them from the original.

## 6. Deviations from the source article

Five of these are defects in the article's design, not preferences.

### 6.1 Dark background baked into every SVG

The article fills text light-gray on a transparent background. That assumes a
dark-mode viewer; on GitHub's **light** theme it renders light-gray on white and
is effectively invisible. Every SVG gets an explicit `#0d1117` rounded background
rect. Fixes the bug and reinforces the terminal-window look.

### 6.2 Split requirements

The article's workflow runs `pip install -r scripts/requirements.txt`, and that
file contains `rembg`, `opencv-python`, and `numpy` — merely *comment-labeled* as
portrait-only. CI therefore downloads roughly 500MB plus an ONNX model **every
day** to run two scripts that need only `requests` and `beautifulsoup4`. Split
into two files; CI installs two packages.

### 6.3 Fail loudly on scrape failure

The contributions endpoint is undocumented and its markup changes periodically.
This is the single largest ongoing risk in the design. The article has no guard,
so a markup change would silently produce an empty heatmap and **commit it over
the good one**. The validation gate in §5.4 instead exits non-zero: the Action
fails, GitHub emails a failure notice, and the last-good SVG stays on the
profile.

### 6.4 Real counts, and five levels not six

The article's `PALETTE` has six entries with a comment describing "level 5 as a
neon top end." GitHub only ever emits `data-level` 0–4, so the sixth entry is
unreachable. Additionally the article implies counts come from the day cells;
they do not — they require the `<tool-tip>` join documented in §5.4.

### 6.5 Honor `prefers-reduced-motion`

Each SVG includes a `@media (prefers-reduced-motion: reduce)` block that jumps to
the final frame instead of animating.

### 6.6 Tunable prep with a preview loop

The article's `prep_photo.py` is fire-once with fixed constants tuned to its
author's lighting. Validated on the actual supplied photo, those settings produce
an unreadable blob. Tone parameters are exposed as CLI flags with a `--preview`
mode, making tuning a fast iterative loop.

## 7. Failure modes

| Failure | Detection | Result |
|---|---|---|
| GitHub changes contributions markup | `<300` cells parsed → exit 1 | Action fails + emails; last-good SVG stays live |
| Endpoint returns non-200 / times out | HTTP status check → exit 1 | Same |
| All cells parse but all levels 0 | Zero-nonzero gate → exit 1 | Same |
| Auto-commit loops on its own push | `[skip ci]` in commit message | No re-trigger |
| SVG renders invisible on light theme | Explicit dark background rect | N/A — prevented |
| Face mirrored or sideways | `exif_transpose` then strip EXIF | N/A — prevented |

---

## 8. Verification criteria

Work is complete when all of the following are confirmed, not assumed:

1. `fetch_contributions.py` writes `data/contributions.json` with 365 day
   entries, and its total matches the profile page.
2. Deliberately corrupting the parse selector causes exit code 1 — the guard is
   proven, not merely written.
3. All three SVGs render legibly against **both** GitHub light and dark themes.
4. The ASCII portrait is recognizable as a face at 370px wide.
5. The rendered profile page shows all three animating on load, then freezing.
6. `workflow_dispatch` run completes green and commits a refreshed SVG.
7. Existing README prose is byte-for-byte unchanged below the header.
